# 同業他社(採用系・RPO・BPO)スクリーニング 設計

## 背景

対象企業リストの中に、人材紹介・人材派遣・RPO(採用代行)・BPO(業務委託)など、自社と同業にあたる企業が混在している。これらは営業対象として不適切なため、リストから除外したい。

`cleanupCompanyData.ts`(表記ゆれ統合)と同様の「一回限りのメンテナンススクリプト」として実装する。今後スプレッドシートに追加される企業を継続的に自動フィルタする仕組みは対象外(スコープ外)。

## 全体フロー

新規スクリプト `scripts/screenCompetitors.ts` として実装する。デフォルトは**ドライラン(候補一覧の出力のみ)**とし、`--apply` を明示的に付けた場合のみスプレッドシートから行を削除する。

```
npm run screen:competitors           # ドライラン
npm run screen:competitors -- --apply # 削除実行
```

`package.json` に以下を追加する。

```json
"screen:competitors": "tsx scripts/screenCompetitors.ts"
```

### ステップ1: 取得

`fetchSheetData` → `parseSheetRows` で全行を読み込む。

### ステップ2: キーワード一覧

新規ファイル `src/lib/competitorKeywords.ts` に判定キーワードを配列で定義する。

```ts
export const COMPETITOR_KEYWORDS = [
  "人材紹介", "人材派遣", "人材バンク", "人材サービス",
  "採用支援", "採用代行", "採用アウトソーシング", "採用マーケティング",
  "RPO", "BPO", "アウトソーシング",
  "ヘッドハンティング", "エグゼクティブサーチ",
  "人事コンサル", "人事コンサルティング",
];
```

(最終的なキーワードは実装時にリストを見ながら微調整して良い。過剰検出は目視確認で弾けるため、多少広めでも可。)

### ステップ3: 判定

行ごとに以下の順で判定し、最初にヒットした根拠を採用する(企業名判定を優先し、ヒットすればページ取得は行わない)。

1. **企業名判定**: `row.companyName` に `COMPETITOR_KEYWORDS` のいずれかが含まれるか。含まれていれば根拠 `{ type: "companyName", keyword }` として候補にする。
2. **ページ内容判定**: 企業名でヒットしなかった行のみ、`row.companyUrl` のトップページ(fetch取得)本文にキーワードが含まれるか確認する。含まれていれば根拠 `{ type: "pageContent", keyword, snippet }` として候補にする(`snippet` はヒット箇所の前後30文字程度、`screen_companies.tmp.mjs` の `findListedKeyword` と同様の方式)。

企業URLが空、または `"なし"` の行はページ取得をスキップする(企業名判定のみ)。

ページ取得は `screen_companies.tmp.mjs` と同じ方式を踏襲する:

- 同時実行数8(`CONCURRENCY = 8`)のワーカープールで処理
- タイムアウト10秒、UAヘッダーを設定して `fetch`
- トップページ本文に加えて、本文中から「会社概要」「会社情報」「企業情報」「事業内容」「事業紹介」「サービス」「About Us」「Company」等のテキストを含むリンク先を1つだけ追加取得し、本文と結合してからキーワード判定する(RPO/BPOの説明はトップページではなく事業内容・サービスページに書かれていることが多いため)
- 取得に失敗した行はエラーとしてカウントし、判定不能として候補には含めない(処理は継続する)

### ステップ4: レポート出力(ドライラン)

コンソールに以下の形式で出力する。

```
=== 同業他社候補: N件 ===
  [行12] 株式会社ABC人材 (https://abc.example.com)
    根拠: 企業名に「人材紹介」
  [行45] 株式会社XYZ (https://xyz.example.com)
    根拠: ページ内容に「RPO」 ("...弊社はRPOサービスを通じて...")
```

候補が0件の場合は「同業他社候補はありませんでした。」と出力する。

### ステップ5: 削除実行(`--apply` 指定時のみ)

- `getSheetId` でシートIDを取得
- 候補行の `rowIndex` 一覧を `deleteRows` に渡して削除する
- 削除件数をコンソールに出力する(`削除しました: N件`)

## エラーハンドリング

- 個別企業のページ取得失敗は候補から除外するのみで、スクリプト全体は継続する(進捗ログに失敗件数を出す、`screen_companies.tmp.mjs` と同様)
- シートAPIエラー(削除失敗)は例外をそのまま投げてスクリプトを終了する(部分適用の可能性があるため、再実行前にスプレッドシートの状態を確認する運用とする。自動リトライは行わない)

## テスト

`tests/competitorKeywords.test.ts` を新規作成し、キーワード判定のロジック(企業名判定・ページ内容判定を行う純粋関数)を単体テストする。ページ取得(fetch)・Sheets API削除は実アクセスを伴うためテスト対象外とし、手動のドライラン結果レビューで代替する(`cleanupCompanyData` と同じ方針)。

## スコープ外

- 今後追加される企業への継続的な自動フィルタ(今回は既存リストの一回限りのクリーンアップ)
- キーワード一覧の完全な網羅性の保証(目視確認で誤検出・見逃しを補う前提)
- 削除した行の復元機能(Google Sheetsの変更履歴による手動復元が前提)
