# フォーム送信ワークフローの高速化 — 設計

## 背景・目的

現状、月1000件のお問い合わせフォーム営業に約17時間かかっている。これを6時間程度に短縮したい。

内訳をヒアリングした結果:
- リスト収集(対象企業・URLの発掘)は既にAIを使って行っており、大きなボトルネックではない
- ボトルネックは「(2) 各フォームを開いて入力・確認する」「(4) 結果をスプレッドシートに手動記入する」の2つ
  - (2)の中でも特に「入力内容の確認」と「次の対象への切り替え」に時間がかかっている

そのため本設計では、リスト収集の自動化(将来別途検討)ではなく、**「対象を開く→入力→[人が確認・送信・キャプチャ対応]→結果記録」の一連の流れをまとめて高速化すること**を対象とする。

既存のCLI(`src/index.ts`)による「1件ずつ処理し`--submit`で自動送信するモード」は本設計の内容に置き換え、廃止する。

## マスターデータ

対象企業リスト・進捗は、これまで運用してきた既存のGoogleスプレッドシートをそのままマスターとして使う(新規にCSVやシートを作らない)。

- シートID: `1-dYD-yUelDgn9PmOQvg-B_7RdcS4cuNRq9ARwJLm2ZU`
- 既存列(抜粋、列名の空白も含めて実際のシートのヘッダーと完全一致させる): リストアップ日 / 企業名 / PRTimes URL / その他URL / 資金調達額 / 企業ラウンド / 企業URL / アマテラス 契約有無 / Notion有無 / フォーム営業 1回目 / フォーム営業 2回目 / フォーム営業 3回目 / 商談 確定日 / 備考 / 資金調達月 / 審査
- 新規追加列: **フォームURL**(発見したお問い合わせフォームのURLをキャッシュする。2回目以降の送信で再探索を避けるため)

認証はGoogleサービスアカウント方式を用いる(対象シートをサービスアカウントのメールアドレスに共有するだけで読み書き可能にする。OAuthのブラウザ認証フローは不要)。サービスアカウントの認証情報ファイルはリポジトリにコミットせず、`.gitignore`で除外する。

## 全体フロー

```
Googleスプレッドシート(マスター)
   │ 対象選定ロジックで「今送るべき行」をN件取得
   ▼
CLIツール(Playwright)
   │ N件分のタブを一気に開く
   │  - フォームURL列があればそれを使う
   │  - なければ企業URLのトップページから
   │    お問い合わせ/contact等のリンクを自動探索して遷移
   │  - 到達したフォームへテンプレート内容を自動入力
   │  - 画面隅に自動入力状況のバナーを表示
   ▼
人手作業
   │ 開いた各タブを順番に確認し、キャプチャ対応・送信ボタンクリック
   │ 全タブ終わったらターミナルでEnterを1回押す
   ▼
CLIツールが各タブの最終状態を判定
   │  成功 / 要確認 / failed を分類
   ▼
Googleスプレッドシートへ結果を書き戻し、タブを閉じる
   │
   ▼ (対象が無くなるまで次のN件へ)
```

## 対象選定ロジック

以下をすべて満たす行をバッチ対象の候補とする:

1. 備考列に既知のスキップ理由(「フォーム無」「Google Formで不可」「電話のみ」「サポートのみ」「リンク切れ」等)が入っていない
2. 「商談」列が「あり」になっていない(既に商談化した企業は対象外)
3. 「フォーム営業3回目」が空欄(3回とも送信済みの企業は対象外)
4. 次の送信タイミングが到来している
   - 「フォーム営業1回目」が空欄 → 初回送信対象
   - 1回目に日付があり2回目が空欄、かつ1回目から30日以上経過 → 2回目(フォローアップ)対象
   - 2回目に日付があり3回目が空欄、かつ2回目から30日以上経過 → 3回目(フォローアップ)対象

候補の先頭からN件(既定20件、CLIオプションで変更可)をバッチとして処理する。

## フォームURLの特定

1. 「フォームURL」列に既に値があればそれをそのまま使う
2. 値がなければ「企業URル」のトップページを開き、リンクテキストやhref に「お問い合わせ/contact/inquiry」等のキーワードを含むリンクを探して遷移する
3. それでも見つからない場合、その行はバッチにカウントせず(タブを開かず)、即座に備考へ「フォーム無(要確認)」を追記して次の候補に差し替える

## 入力内容確認の負担軽減

各タブを開いて自動入力した際、画面の隅に小さなバナーを自動挿入し、テンプレートの各項目が入力できたかどうかを一目で分かるようにする。
例: 「自動入力: 会社名○ 氏名○ メール○ 電話✗(未検出)」
未検出項目がある場合はそこだけ人が手で入力すればよく、フォーム全体を目で追う必要がなくなる。

## 送信成功判定ロジック(バッチ終了・Enter押下後にチェック)

各タブについて、以下のいずれかを満たせば「成功」、どれも満たさなければ「要確認」:
- 元のURLからページ遷移している(URLが変わった)
- ページ内に「ありがとうございました」「送信が完了」「受け付けました」等の完了を示す文言がある
- 元のフォーム要素がページから消えている

## 結果の書き込み

- **成功** → 該当する「フォーム営業N回目」列(今回が何回目の送信だったか)に実行日を記入。フォームURLを新規発見した場合は「フォームURL」列にも記入
- **要確認**(判定できず) → 「フォーム営業N回目」列に実行日を記入し、かつ備考に「要確認」を追記する。実行日を記入するのは、二重送信を防ぐことを優先するため。あとで人が備考を見て該当企業のフォーム送信結果を確認できる
- **failed**(ページ到達不可・フォーム未発見など) → 「フォーム営業N回目」列には記入しない(未実施扱いとし、次回バッチで再挑戦されるようにする)。備考に失敗理由を追記する

「返信あり」「商談化」など、送信後の反応に関するステータスは今回の自動化の対象外。これらは引き続き人が備考・商談確定日列に手動で記入する運用のままとし、スクリプトはこれらの列を読み書きしない。

## エラーハンドリング

- 企業URL/フォームURLへの到達失敗(タイムアウト・DNSエラー等) → failed扱い。備考に理由を追記、フォーム営業N回目列は空のまま次回リトライ
- お問い合わせリンクが見つからない → 「フォーム無(要確認)」を備考に追記し、バッチにはカウントしない
- Google Sheets APIへの書き込み失敗 → その場でエラー表示し、結果をローカルに一時保存(取りこぼし防止)。次回実行時に再送できるようにする

## スコープ外(今回は対象としない)

- リスト収集(対象企業・URLの新規発掘)の自動化 — 別途ブレストする
- 既存の他スプレッドシートからの一括インポート — 発生頻度が低いため手動コピペで対応する
- 「返信あり」「商談化」等、送信後の反応トラッキングの自動化

## テスト方針

- 対象選定ロジック(備考のスキップ判定・日数境界の判定)のユニットテスト
- フォームURL自動探索・入力・送信成功判定ロジックは、実際のフォームHTMLサンプルを使ったユニットテスト
- `--headed`でのバッチ実行を少数件で手動確認し、実際の使用感を確認する

## 既存コードへの影響

- `src/index.ts` の「1件ずつ処理・`--submit`で自動送信」フローは削除し、本設計のバッチフローに置き換える
- `data/targets.csv` / `data/results.csv` およびそれらを扱う `src/lib/targets.ts` / `src/lib/results.ts` はGoogleスプレッドシート連携に置き換わるため、役目を終える(削除または置き換え)
- `src/lib/formSubmitter.ts` の自動入力ロジック(キーワードベースのフィールド検出)は流用する
