# スプレッドシート企業データ クリーニング 設計

## 背景

現状のスプレッドシート(1454行)には、企業名などのテキストに前後の余分な空白・全角スペース・ゼロ幅スペースなどの不可視文字が混入している行や、表記ゆれ(「Luup」と「株式会社Luup」など)により同一企業が複数行に分かれているケースがある。実行時の`dedupeByCompanyName`(`src/lib/targetSelection.ts`)は前後空白・大文字小文字の差異のみを吸収するため、法人格の有無・位置が違う表記ゆれまでは統合できず、同じ企業に重複してフォーム営業してしまうリスクが残っている。

これは以下3つの独立した作業のうち①にあたる(②非スタートアップ企業の除外、③2026年以降の新規資金調達企業の追加は別途設計する)。

## スコープ

- 対象列: 企業名・企業URL・フォームURL・備考の全列(値がある全セル)
- 対象外: 商談確定日・フォーム営業1〜3回目(日付形式が決まっている列のため触らない)

## 全体フロー

一回限りのメンテナンススクリプト`scripts/cleanupCompanyData.ts`として実装する。既存の`sheetsClient.ts`/`sheetData.ts`の関数を再利用し、**ドライラン(レポート出力のみ)**と**実書き込み**の2モードを持つ(`--apply`フラグの有無で切り替え)。デフォルトはドライランとし、事故防止のため明示的に`--apply`を付けない限り書き込みは一切行わない。

```
scripts/cleanupCompanyData.ts --template <report出力先> [--apply]
```

### ステップ1: 取得

`fetchSheetData` → `parseSheetRows`で全行を読み込む。

### ステップ2: テキスト正規化

新規関数`normalizeCellText(value: string): string`(`src/lib/textNormalize.ts`)を作成する。

```ts
export function normalizeCellText(value: string): string
```

- 前後の半角/全角空白をトリム
- ゼロ幅スペース(U+200B)・ゼロ幅接合子(U+200C/200D)・BOM(U+FEFF)を除去
- 内部の連続する全角/半角空白は1つの半角空白に正規化
- 上記以外の文字は変更しない(絵文字・記号等はそのまま)

企業名・企業URL・フォームURL・備考の各セルにこの関数を適用し、変更前後で値が異なる行を「正規化対象リスト」として記録する。

### ステップ3: 重複候補の抽出(コア名比較)

新規関数`extractCompanyCoreName(name: string): string`(同ファイル)を作成する。

```ts
export function extractCompanyCoreName(name: string): string
```

- 正規化後の企業名から法人格トークン(`株式会社`、`㈱`、`有限会社`、`合同会社`、`Inc.`、`Inc`、`Co., Ltd.`、`Co.,Ltd.`、`Ltd.`、`Ltd`、`K.K.`、`Corporation`、`Corp.`)を(前後どちらの位置でも)除去
- 残った文字列から記号・スペースを除去し、小文字化して返す

正規化後の全行を`extractCompanyCoreName`の結果でグループ化する。同じコア名を持つ行が2行以上あるグループを「重複候補グループ」とする。

### ステップ4: Fable 5による最終判定

重複候補グループのうち、**正規化後の企業名が完全一致ではないペア**(コア名は一致するが表記が異なる組み合わせ)について、Agentツールに`model: "fable"`を指定して以下を判定させる。

- 入力: 2社分の企業名・企業URL(あれば)
- 出力: `{ sameCompany: boolean, confidence: "high" | "low", reason: string }`
- `confidence: "low"`、またはAgent呼び出し自体が失敗した場合は自動統合せず「要目視確認」としてレポートに残す

正規化後の企業名を前後空白トリム・大文字小文字を無視して比較し、完全一致するペア(表記ゆれではなく実質的に同じ文字列)は、Fableに問い合わせず自動的に「同一企業」として扱う(既存の`dedupeByCompanyName`と同じ比較基準)。

コア名が一致する行が3つ以上あるグループでは、完全一致しない組み合わせすべてをFableに判定させる。一部のペアが「同一企業でない」と判定された場合、そのグループ全体を自動統合せず「要目視確認」としてレポートに残す(部分的にのみ一致する場合の誤統合を避けるため)。

### ステップ5: 統合対象の決定

`sameCompany: true`かつ`confidence: "high"`と判定されたグループについて、既存の`dedupeByCompanyName`と同じ優先順位ロジック(商談確定日が入っている行 > フォーム営業3回目 > 2回目 > 1回目が入っている行を優先し、同点なら元の行順で先頭を残す)で「残す行」を1つ決定し、それ以外を「削除対象行」とする。

### ステップ6: レポート出力

ドライラン時は以下をコンソールに出力するのみで、シートへの書き込みは行わない。

- 正規化によって値が変わるセル一覧(行番号・列名・変更前後の値)
- 統合により削除される行一覧(行番号・企業名・削除理由=統合先の行番号)
- 「要目視確認」となったペア一覧(Fableの判定理由つき)

### ステップ7: 実書き込み(`--apply`指定時のみ)

- 正規化対象リストの値を`writeCells`で書き込む
- 削除対象行を`deleteDimension`(Sheets API `batchUpdate`)で削除する。同一シート内で複数行を削除する場合、行番号が大きい順に削除リクエストを並べて1回の`batchUpdate`に含める(先に大きい行番号を消すことで、後続の削除対象の行番号がずれない)

## エラーハンドリング

- Fable判定のAgent呼び出しが例外を投げた場合、そのペアは「要目視確認」としてレポートに残し、処理全体は継続する(1件の失敗で他の判定を止めない)
- シート書き込み・行削除でAPIエラーが発生した場合は、どこまで反映されたかをコンソールに出力してスクリプトを終了する(部分適用の状態を明示する。自動リトライは行わない)

## テスト

`tests/textNormalize.test.ts`を新規作成し、以下をユニットテストする(Fable判定・Sheets API書き込みは実アクセスを伴うためテスト対象外、手動のドライラン結果レビューで代替する):

- `normalizeCellText`: 前後の半角/全角空白除去、ゼロ幅スペース・BOM除去、内部の連続空白の正規化、通常のテキストは変更されないこと
- `extractCompanyCoreName`: 法人格トークンが前後どちらにあっても除去されること、記号・大文字小文字の差異を吸収すること、法人格トークンを含まない社名はそのまま(小文字化のみ)であること

## スコープ外

- ②非スタートアップ企業の判定・除外(別設計)
- ③2026年以降の新規資金調達企業の発掘・追加(別設計)
- 企業名以外の基準(例: 企業URLのドメイン一致)での重複検出
- 削除した行の復元機能(Google Sheetsの変更履歴(バージョン管理)による手動復元を前提とする)
